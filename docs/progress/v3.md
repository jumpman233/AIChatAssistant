# V3-1：后端并发边界与 Conversation DTO

本步骤只处理后端，不修改前端。

## 目标

保证：

* 不同 conversation 可以同时 streaming
* 同一 conversation 只能有一个 active assistant stream
* A 的重复请求返回 409
* A 的 409 不影响 A 原 stream 和 B stream
* list/detail 能反映真实 streaming 状态

## 任务

### 1. 检查 active guard 并发安全

如果当前流程只是：

```text
查询 active message
-> 没查到
-> 创建 user / assistant message
```

两个同会话请求可能同时通过，需要做最小并发安全修复。

允许使用：

* PostgreSQL 行锁
* Prisma transaction
* 合理的事务隔离
* 其他数据库级最小方案

禁止：

* 用进程内全局 Map 作为唯一锁
* Redis
* 分布式锁
* 队列
* 阻塞不同 conversation

目标是锁定单个 conversation，而不是锁整个聊天服务。

### 2. 保持 409 契约

同会话已有 active assistant 时：

```text
HTTP 409
code = CONVERSATION_STREAMING
```

错误结构遵守现有 API 契约。

### 3. Conversation DTO

streaming 期间：

```text
isStreaming = true
activeAssistantMessageId = 当前 active assistant id
```

完成或失败后：

```text
isStreaming = false
activeAssistantMessageId = null
```

A、B 必须分别返回自己的状态。

## 本步骤验证

必须运行：

```bash
pnpm typecheck
pnpm verify:v1
pnpm verify:v2
```

可以做一个轻量人工或临时 smoke：

* 使用较慢 Mock Stream
* 并发请求 A、B
* 确认都能启动
* 再请求 A，确认 409
* 不要求此时正式新增 `verify:v3`

完成后输出：

1. 是否发现 guard 竞态。
2. 如何修复。
3. 是否使用了数据库级并发边界。
4. DTO streaming 状态如何计算。
5. 验证结果。
6. `git diff`。

# V3-2：前端请求与 Runtime 隔离

本步骤处理前端核心运行时，不先做完整侧边栏 UI。

## 目标

确保每个 stream 从发起到完成，始终绑定自己的 conversationId，不受当前页面切换影响。

## 任务

### 1. 固定请求上下文

每次 `sendChat` 开始时固定保存：

```text
conversationId
AbortController
reader / request
streamId
assistantMessageId
runtime state
```

所有异步回调必须使用这次请求绑定的 conversationId。

禁止在这些回调里使用当前：

```text
activeConversationId
```

来决定更新哪个 conversation。

### 2. conversationStore 隔离

确认或补齐：

* `messagesByConversationId`
* A 的 message 只能进入 A
* B 的 message 只能进入 B
* `message_created` 不重复插入
* `text_delta` 更新正确 assistant
* `message_done` 更新正确 assistant
* `message_failed` 更新正确 assistant
* 一个 conversation 完成不能覆盖另一个 conversation 的状态

### 3. chatRuntimeStore 隔离

每个 conversation 独立保存：

* `isStreaming`
* `streamingMessageId`
* `AbortController`
* reader / request
* error
* typewriter state
* timer

要求：

* A、B runtime 可同时存在
* A 完成只清理 A
* B 完成只清理 B
* 页面整体卸载仍能统一清理全部 runtime
* 切换页面不清理后台 conversation runtime
* V3 不新增停止按钮

### 4. typewriter 隔离

确认：

* A 和 B 使用独立 pendingText
* A 的 timer 不消费 B 的内容
* MessageItem 卸载不销毁 store 中仍在运行的 typewriter
* 切回 A 后能继续看到 A 的 displayContent
* 一个 message done 后只清理该 message runtime

## 本步骤验证

运行：

```bash
pnpm typecheck
```

人工验证优先，不要求此时写新脚本：

1. 用 `pnpm dev:mock` 启动。
2. 配置较明显的 Mock delay。
3. 在 A 发消息。
4. A 未完成时切到 B。
5. 在 B 发消息。
6. 多次切换 A/B。
7. 确认内容不串。
8. 确认 A 完成不影响 B。
9. 刷新后确认最终消息正确恢复。

完成后输出：

1. 哪些异步代码原来依赖 activeConversationId。
2. 如何绑定固定 conversationId。
3. runtime 如何隔离。
4. 人工验证结果。
5. `git diff`。

# V3-3：侧边栏状态与发送交互

本步骤只补 UI 状态，不做复杂视觉还原。

## 目标

用户能看出哪些会话正在生成，并且一个会话生成时不影响其他会话发送。

## 任务

### 1. 左侧会话列表

ConversationItem / Sidebar 显示：

* 当前会话 streaming
* 非当前会话 streaming
* 完成后只清除对应会话标识

标识可以是：

* spinner
* generating 文案
* 小状态点

不要求复杂动画。

### 2. 输入框禁用逻辑

判断目标 conversation：

```text
isConversationStreaming(conversationId)
```

而不是全局：

```text
isAnyConversationStreaming
```

要求：

* A streaming 时，A 不能再次发送
* A streaming 时，B 可以发送
* B streaming 时，不影响新建或其他非 streaming 会话
* 后端 409 仍是最终权威

### 3. ConversationDTO 本地同步

收到：

* `message_created`
* `message_done`
* `message_failed`

时，本地 conversation 状态应同步更新。

最终状态仍要与后端 list/detail 一致。

## 本步骤验证

运行：


人工验证：

1. A 发消息。
2. 切 B 发消息。
3. 左侧 A、B 同时显示 generating。
4. A 结束后只清除 A 标识。
5. B 继续显示。
6. 尝试在 A streaming 时再次发送，UI 应阻止。
7. B 仍可正常发送。

完成后输出：

1. 侧边栏状态如何判断。
2. 输入框如何按 conversationId 禁用。
3. 是否存在全局禁用问题。
4. 人工验证结果。

## Mock 延迟

使用足够长但不过分拖慢测试的 delay，确保 A、B 真正重叠。

不要只因为使用了 `Promise.all` 就认定发生并发。

## 执行流程

1. 创建 A、B。
2. 同时发起 A、B `/api/chat`。
3. 分别读取两个 stream。
4. 确认 A、B 都收到 `message_created`，且尚未 done。
5. 此时再次请求 A。
6. 断言：

   * HTTP 409
   * `code = CONVERSATION_STREAMING`
7. streaming 期间查询 list/detail。
8. 等待 A、B 完成。
9. 完成后再次查询 list/detail。
10. 查询 messages API。
11. 查询数据库。

## 必须断言

### 时间重叠

至少证明：

```text
A 已 message_created 且未 done
B 已 message_created 且未 done
```

### Event 隔离

* A event conversationId 全部是 A
* B event conversationId 全部是 B
* A/B streamId 不同
* A/B assistantMessageId 不同
* A/B text_delta messageId 各自正确
* event 与 data.type 一致

### 409 不影响原流

A duplicate 返回 409 后：

* A 原 stream 继续完成
* B stream 继续完成
* 两个 assistant 最终都是 done
* 不出现 failed / aborted

### Streaming DTO

streaming 期间：

* A、B `isStreaming=true`
* A、B `activeAssistantMessageId` 各自正确
* A/B activeAssistantMessageId 不同

完成后：

* A、B `isStreaming=false`
* A、B `activeAssistantMessageId=null`

### API / DB

* A content 等于 A delta 拼接
* B content 等于 B delta 拼接
* 不跨 conversation 污染
* 每个 conversation 内 seq 唯一
* user / assistant 均落库
* assistant 均 done
* toolCalls 是数组

## 日志

输出安全摘要：

```text
[verify:v3] streams overlapped
[verify:v3] duplicate A rejected
[verify:v3] A events: message_created -> text_delta xN -> message_done
[verify:v3] B events: message_created -> text_delta xN -> message_done
[verify:v3] streaming DTO checked
[verify:v3] API/DB isolation checked
[verify:v3] passed
```

不输出完整 prompt 和完整回答。
